## 7) Addendum 2026-03-06: Hotspot Unauth + Stale IP Hardening

Scope addendum ini menutup gap flow captive saat `client_ip`/`client_mac` hilang atau stale pada jalur hotspot-required dan auto-login.

Temuan root-cause dari log produksi (06-03-2026):
- `POST /api/auth/verify-otp` dari halaman `/login` berhasil (`200`) lalu user langsung diarahkan ke `/dashboard`.
- Redirect ke `hotspot-required` saat itu bergantung pada syarat `hasHotspotIdentity || hasHotspotLoginLink`, sehingga login biasa tanpa hint query bisa "tembus" dashboard lebih dulu.
- Setelah masuk dashboard, endpoint best-effort `/users/me/devices/bind-current?best_effort=1` baru dieksekusi; urutan ini tidak sesuai UX/policy yang diharapkan.

Perubahan kode utama:
- Frontend menambah resolver identitas terpadu di `frontend/utils/hotspotIdentity.ts` dengan urutan sumber `query -> referrer -> sessionStorage`, TTL context 10 menit, serta normalisasi MAC.
- `frontend/pages/login/hotspot-required.vue` dan `frontend/store/auth.ts` kini memakai resolver identitas terpadu untuk status polling, one-click authorize, dan bootstrap auto-login.
- `frontend/pages/login/index.vue` kini:
  - memprioritaskan user demo langsung ke `/beli`,
  - selalu mengevaluasi kebutuhan `hotspot-required` (tanpa mensyaratkan query hint),
  - melakukan fallback check ke `/auth/hotspot-session-status` sebelum redirect final ke dashboard.
- `frontend/middleware/auth.global.ts` kini:
  - menjalankan hotspot precheck juga pada route `/dashboard` untuk user non-admin,
  - memprioritaskan redirect user demo ke `/beli` sebelum hotspot precheck.
- `frontend/pages/captive/index.vue` menyimpan context identitas lebih awal agar lintas redirect tidak kehilangan hint.
- Backend `backend/app/infrastructure/http/auth_contexts/hotspot_status_handlers.py` menambah guard `HOTSPOT_SESSION_STATUS_REQUIRE_IDENTITY_HINT=True` (default), derivasi MAC dari hotspot user IP saat query kosong, serta menonaktifkan lookup user-level tanpa identitas (anti false-positive stale binding).
- Backend `backend/app/infrastructure/http/auth_contexts/login_handlers.py` menambah fallback recovery IP dari router via `get_ip_by_mac` saat IP hint stale, tetap dengan validasi CIDR dan MAC router-authoritative.

Validasi test addendum:

- Frontend: `npm run test -- tests/hotspot-identity.test.ts tests/hotspot-redirect.test.ts` -> `2 files passed`, `11 tests passed`.

## 8) Addendum 2026-03-06: Incident DB Kosong Berulang (Root Cause + Mitigasi)

Temuan produksi (06-03-2026):
- DB kembali kosong total (`users=0`, `user_login_history=0`, `transactions=0`).
- Data dipulihkan kembali dari backup sehat: `_safe_backups/deploy_predeploy_20260306_034620/postgres_dump.sql`.
- Verifikasi pasca restore: `users=98`, `user_login_history=140`, `transactions=5`.

Root-cause paling mungkin:
- Task scheduler Celery `clear_total_if_no_update_submission_task` memang menjalankan:
  - `TRUNCATE TABLE users RESTART IDENTITY CASCADE`
- Dengan konfigurasi produksi saat insiden:
  - `UPDATE_ENABLE_SYNC=True`
  - `UPDATE_CLEAR_TOTAL_AFTER_DAYS=3`
  - `public_database_update_submissions.latest_submission` = `2026-03-02` (stale terhadap hari insiden)
- Karena `CASCADE`, tabel relasi ikut kosong (termasuk login history dan transaksi yang bergantung FK ke users).

Mitigasi langsung (ops):
- Runtime `.env.prod` produksi diubah ke `UPDATE_ENABLE_SYNC=False`.
- Backend + Celery worker/beat direcreate agar scheduler destruktif berhenti.

Mitigasi permanen (kode):
- Tambah hard guard baru `UPDATE_ALLOW_DESTRUCTIVE_AUTO_CLEAR` (default `False`).
- Task auto-clear sekarang tidak akan jalan walau `UPDATE_ENABLE_SYNC=True`, kecuali guard destruktif ini diaktifkan eksplisit.
- Registrasi schedule `clear-total-if-no-update-submission` juga kini mensyaratkan guard destruktif aktif.

File terkait mitigasi permanen:
- `backend/config.py`
- `backend/app/tasks.py`
- `backend/app/extensions.py`
- `backend/tests/test_tasks_clear_total_auto_clear.py`
