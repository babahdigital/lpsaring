## 1. Root Cause Yang Diselesaikan (Lanjutan dari 2026-03-08)

### Problem: WA tidak pernah terkirim ke user Imported

Tabel `public_database_update_submissions` selalu kosong karena tidak ada mekanisme otomatis yang mengisi stub dari data User. WA batch task (`send_public_update_submission_whatsapp_batch_task`) hanya memproses data yang sudah ada di tabel ini.

### Solusi: Task populate otomatis

**`populate_update_submissions_from_imported_users_task`** — scan tabel `User` untuk nama `LIKE 'Imported %'`, buat stub di `public_database_update_submissions` jika belum ada.

### G) Bug fix: Stub tampil di admin panel

**`backend/app/infrastructure/http/admin/user_management_routes.py`** — tambah filter `source_ip != "system:populate_task"` pada query `list_public_update_submissions`:

```python

    PublicDatabaseUpdateSubmission.approval_status == status,
    PublicDatabaseUpdateSubmission.source_ip != "system:populate_task",  # ← FIX

```

**Desain yang benar:**
- Stub (`source_ip=system:populate_task`) → backend only, untuk WA sending + status check
- Human submission (real `source_ip`) → tampil di admin panel untuk direview

## 8. Analisa Nginx Error Log (2026-03-07 — 2026-03-09)

### Ringkasan error per tipe

| Tipe Error | Jumlah | Penyebab | Status |
|---|---|---|---|
| `lpsaring-backend could not be resolved (2: Server failure)` | ~40 | DNS Docker gagal saat backend container restart/recreate | Pre-deploy (07-08 Mar) |
| `upstream prematurely closed connection` | 7 | Backend worker menutup koneksi (Gunicorn restart / DB query lambat) | Pre-deploy |
| `connect() failed (111: Connection refused)` | 4 | Backend belum ready setelah restart | Pre-deploy |
| `upstream timed out (110: Operation timed out)` | 5 | Nuxt SSR lambat untuk Android captive portal probe | Pre-deploy |
| `recv() failed (104: Connection reset by peer)` | 2 | Koneksi TCP drop saat backend restart | Pre-deploy |

**Post-deploy hari ini (09 Mar 05:43+): ZERO error.**

### Analisa DNS error (`could not be resolved`)

**Root cause:** Nginx menggunakan `resolver 127.0.0.11 valid=10s` + `set $backend_host` pattern. Saat backend container di-recreate:
1. Container lama stop → Docker DNS tidak lagi resolve `lpsaring-backend`
2. Container baru start → Docker DNS butuh ~5–30 detik untuk register
3. Selama window ini, semua request ke `/api/` gagal dengan DNS error

**Perbaikan yang sudah dilakukan:**
- `valid=5s` (dari 10s) → cache DNS lebih pendek, deteksi lebih cepat
- `proxy_connect_timeout 10s` + `proxy_next_upstream error timeout http_502 http_503 http_504` → failover otomatis

**Window error yang tersisa:** ~5–30 detik saat deploy (unavoidable tanpa load balancer). Bisa dikurangi lebih lanjut dengan:
```nginx
# Tambahkan keepalive + health check di upstream block
upstream lpsaring_backend {
    server lpsaring-backend:5010;
    keepalive 10;
}
```
Ini opsional — dengan `--recreated` deploy yang baru, window downtime sudah sangat minimal.

### Analisa Nuxt SSR timeout

`upstream timed out` pada frontend (192.168.0.6:3010) terjadi di:
- `06:35` dan `06:37` — Android user (202.65.239.46) hit `/`
- `12:06` — Android user hit `/login`
- `16:05`, `16:06`, `16:06` — short cluster

**Penyebab:** Nuxt SSR di Raspberry Pi kadang lambat untuk render kompleks saat CPU/memory ada tekanan dari task lain (sync MikroTik). Ini bukan masalah kritis — Android menganggap semua non-204 response sebagai captive portal respon yang valid.

**Mitigasi yang sudah ada:**
- `proxy_send_timeout 15s` ditambah ke semua location
- `proxy_next_upstream_timeout 5s` + `proxy_next_upstream_tries 2`
