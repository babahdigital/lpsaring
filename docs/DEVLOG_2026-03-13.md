## Bug 1: bind-current SIGKILL dari Dashboard

### Gejala
- 2 kali SIGKILL di jam 12:09 dan 12:10 WIB (04:09–04:10 UTC)
- Nginx: `upstream prematurely closed` → `POST /api/users/me/devices/bind-current?best_effort=1`
- Backend: `Worker was sent SIGKILL! Perhaps out of memory?` (padahal RAM 4.1 GB available → bukan OOM, tapi timeout)

### Root Cause
`apply_device_binding_for_login()` tanpa `client_ip` (panggilan dari dashboard) membuka 3 koneksi MikroTik TCP berurutan:
1. `_resolve_binding_ip` → cari IP user via hotspot entry atau ARP
2. `resolve_client_mac` (dalam `register_or_update_device`) → cari MAC dari IP via ARP
3. `_apply_post_auth_mikrotik_ops` → ip-binding + address-list + hotspot host cleanup

Masing-masing koneksi bisa 60–70s di worst case → total ~180s >> timeout gunicorn 120s → SIGKILL → 502.

### Fix (`76e66bca`)
- Tambah `api_connection: Optional[Any] = None` ke 4 fungsi: `_resolve_binding_ip`, `resolve_client_mac`, `register_or_update_device`, `_apply_post_auth_mikrotik_ops`
- Di `apply_device_binding_for_login`: deteksi "slow path" (client_ip tidak valid) → buka **satu koneksi bersama** via `nullcontext(None)` / `get_mikrotik_connection()` → teruskan ke semua fungsi sebagai `shared_api`
- Fast path (captive portal dengan IP valid): tidak berubah — `shared_api=None`, fungsi buka koneksinya sendiri (total 1 koneksi untuk post-auth ops)

### Dampak
- Dashboard bind-current: 3 koneksi → **1 koneksi** (~60s durasinya, jauh di bawah 120s timeout)
- Captive portal: tidak berubah (sudah fast path, 1 koneksi)

## Bug 2: hotspot-required.vue "Aktifkan Internet" Tidak Berfungsi

### Gejala
Setelah OTP berhasil dan `authorizeDevice()` dipanggil, tombol "Aktifkan Internet" tidak lanjut ke portal MikroTik — 8 poll `hotspot-session-status` semua return `binding_active=False`.

### Root Cause
User `bypassed` (mis. user unlimited) TIDAK masuk ke `/ip/hotspot/host` MikroTik karena bypass portal.
`hotspot-session-status` dengan `lookup_mode=router-mac-from-user-ip` mencari entry di hotspot host → tidak ada → `missing-hints` → `binding_active=False`.

### Fix (`c39dcd0b`)
- **Fix A (backend)**: tambah `db-device-mac` fallback di `hotspot_status_handlers.py`:
  - Jika hotspot host tidak ada, ambil MAC dari DB `UserDevice` terbaru user
  - Cek ip-binding MikroTik berdasarkan MAC tersebut
  - Jika ip-binding ada (tipe `bypassed` / `regular`) → return `binding_active=True`
- **Fix B (frontend)**: jika `authorizeDevice()` return `true` → polling diperpendek: 3× (bukan 8×), interval 900ms → setelah itu langsung `continueToPortal()` tanpa tunggu hotspot host entry

## Bug 3: Webhook Payment Loss (3 Bug Terkait)

### Konteks
User +6285166001405 bayar Rp 100.000 QRIS tapi app tampilkan EXPIRED. Investigasi menemukan 3 bug:

### Bug 3a: Generator Double-Yield (`mikrotik_client.py`)
`yield None` di dalam `try:` block menyebabkan `RuntimeError: generator didn't stop after throw()` saat caller raise exception. Fix: acquire connection BEFORE yield, pindah semua failure yield ke luar try/except.

### Bug 3b: Idempotency TTL Terlalu Panjang (`idempotency.py`)
Redis dedup key di-set saat webhook start dengan TTL 86400s (24 jam). Jika worker SIGKILL sebelum DB commit (gunicorn timeout 120s), key tetap ada → retry webhook diblokir 24 jam → pembayaran tidak bisa diproses ulang. Fix: TTL 86400s → 30s (hanya untuk burst dedup).

### Bug 3c: Nested MikroTik Connections di Webhook
`sync_address_list_for_single_user` (dipanggil dari `transaction_service.py`) membuka koneksi MikroTik EKSTRA padahal sudah ada koneksi terbuka di caller → nested TCP overhead → total >120s → SIGKILL → DB rollback → payment hilang. Fix: `api_connection=None` param, pass koneksi existing dari caller.

## Infrastruktur Hardening (`e4129451`)

### MikroTik TCP Connection Leak
`mikrotik_client.py`: `finally` block tidak memanggil `disconnect()` jika pool tidak support `return_api` → socket table MikroTik cepat penuh. Fix: eksplisit `disconnect_fn()` di `finally`.

### Redis Memory Limit
`maxmemory=0` (unlimited) → OOM risk saat Redis tumbuh. Fix: `--maxmemory 256mb --maxmemory-policy allkeys-lru`.

### SQLAlchemy Stale Connection
`pool_pre_ping=False`, `pool_recycle=-1` default → koneksi stale setelah PostgreSQL timeout 8 jam → query error. Fix: `pool_pre_ping=True`, `pool_recycle=3600`, `pool_size=3`, `max_overflow=5` di ProductionConfig.

### Gunicorn Timeout
`--timeout 30` → `--timeout 120`, tambah `--graceful-timeout 120`, `--keep-alive 2`.

## Bug 4: FUP Stale Address-List saat IP Recycling (`aed7b3c7`)

### Gejala
User +6283844097553 muncul di parity dashboard dengan mismatch:
`active / active | bypassed / bypassed | fup | address_list` — ada entry `klient_fup` di MikroTik
padahal user sudah di-binding ulang saat login.

### Root Cause
`_apply_post_auth_mikrotik_ops` (langkah 2) hanya menghapus IP dari `klient_blocked` dan
`klient_unauthorized` saat login. Ketika DHCP mendaur-ulang IP dari user FUP lama ke user baru,
entry `klient_fup` / `klient_habis` / `klient_expired` / `klient_inactive` dari user lama tetap
ada → parity guard mendeteksi mismatch, user baru tampak FUP di MikroTik walau belum pernah
menghabiskan kuota.

### Fix
Tambah `_STALE_LISTS_TO_CLEAR` yang mencakup 6 list status:
`{list_blocked, list_unauthorized, list_fup, list_habis, list_expired, list_inactive}`.
Step 2 di `_apply_post_auth_mikrotik_ops` kini loop hapus IP dari semua 6 list tersebut.

`klient_aktif` **tidak** dibersihkan — user yang memang aktif harus tetap ada.
Jika user sedang FUP, periodic sync (maks ~2 menit) akan re-add ke `klient_fup` otomatis.

### Catatan Investigasi +6283844097553
- Kuota sisa: `2798 MB < 3072 MB threshold` → legitimately FUP saat ini
- Gap Celery 71 menit (04:41–05:52 UTC) saat deploy → 341 MB tidak ter-track → FUP crossing terdeteksi sekaligus
- MAC `0A:2A:AC:00:9B:91` adalah locally-administered (MAC randomized) → kemungkinan besar mismatch awalnya dari IP recycling

## Bug 5: MikroTik socket.recv() Hang — SIGKILL Persists Post-Deploy (`273f0f95`)

### Gejala (Post `76e66bca`)
4 SIGKILL masih terjadi setelah fix konsolidasi koneksi: 07:31, 09:22 (×2), 09:49 UTC. Worker
mati tepat 120s setelah pool diinisialisasi. Cascade: PostgreSQL "too many clients" jam 09:51 UTC
(timeout TCP connections dari worker yang di-SIGKILL tidak di-cleanup segera).

### Root Cause — routeros_api 0.21 Architecture

Setelah analisis source code PyPI `routeros_api 0.21.0`:

```
# RouterOsApiPool.__init__() — tidak ada socket_timeout param
(host, username, password, port, plaintext_login, use_ssl, ssl_verify, ssl_verify_hostname, ssl_context)

# socket_timeout = CLASS ATTRIBUTE, default 15.0 detik
# Berpengaruh saat get_api() dipanggil:
def get_api(self):
    if not self.connected:
        self.socket = api_socket.get_socket(host, port, timeout=self.socket_timeout, ...)
        # socket.create_connection((host, port), timeout=15.0)
        # → socket mempertahankan timeout ini untuk SEMUA operasi recv/send
```

Kritikalnya: `socket.create_connection((host, port), timeout=N)` menjadikan socket non-blocking
dengan timeout N untuk **semua** operasi termasuk `recv()`. Jika class default 15.0 tetap dipakai,
`resource.get()` membaca response dari MikroTik selama 15s sebelum timeout — tapi gunicorn
membunuh worker di 120s, bukan 15s. Penyebab sebenarnya: `socket_timeout` belum dioverride.

### Fix Salah: `c460aa4f` — `_try_set_socket_timeout(api_instance, ...)`

Mencoba set timeout via attribute paths pada `api_instance`:
```python
_attr_paths = [("base_api", "socket", "socket"), ("socket", "socket"), ...]
```
**SALAH** — `api_instance` adalah `RouterOsApi(communicator)` yang hanya punya `.communicator`.
Tidak ada `base_api`, tidak ada `socket` di top-level. Semua 4 path gagal silently → tidak ada efek.

### Fix Benar: `273f0f95` — Dua Pendekatan Redundan

**Pendekatan 1** — `init_mikrotik_pool()`: set instance attribute SEBELUM `get_api()` pertama:
```python
_connection_pool = routeros_api.RouterOsApiPool(host, **pool_kwargs)
if not supports_socket_timeout:
    _connection_pool.socket_timeout = socket_timeout_seconds  # override class default 15.0
```
Efek: `pool.get_api()` akan `socket.create_connection(..., timeout=10.0)` → socket recv timeout = 10s.

**Pendekatan 2** — `get_mikrotik_connection()` via `_apply_pool_socket_timeout()`:
```python
def set_timeout(self, socket_timeout):
    self.socket_timeout = socket_timeout
    self.socket.settimeout(socket_timeout)  # SocketWrapper → raw socket.settimeout()
```
Dipanggil SETELAH `_get_api_with_timeout()` sukses → update live socket jika pool sudah connected.

**Keduanya benar dan saling melengkapi**:
- Pendekatan 1 fix connect yang belum terjadi (pool baru dibuat)
- Pendekatan 2 fix socket yang sudah connected (pool reconnect otomatis)

### Catatan routeros_api 0.21 Architecture
```
RouterOsApiPool
├── socket_timeout = 15.0              ← CLASS ATTRIBUTE (bukan constructor param)
├── socket = DummySocket()             ← sebelum connect
├── get_api()                          ← connect + return RouterOsApi
│     └── socket = SocketWrapper(raw)  ← setelah connect, recv timeout = socket_timeout
├── set_timeout(t)                     ← set socket_timeout + socket.settimeout(t)
└── disconnect()                       ← reset socket ke DummySocket

RouterOsApi (returned by get_api())
└── communicator                       ← satu-satunya attribute relevan
```
