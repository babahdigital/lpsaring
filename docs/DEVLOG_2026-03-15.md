## Context

Sesi ini menutup dua gap yang masih tersisa setelah rollout frontend/backend sebelumnya:

1. direct-browser OTP masih bisa jatuh ke `/login/hotspot-required` dengan `identity_hint_missing`,
2. user yang ter-block otomatis oleh `policy.block_transition:sync_usage` karena `quota_debt_limit` belum menerima WA khusus debt-limit.

Selain implementasi kode, sesi ini juga mencakup push, CI, docker publish, deploy, verifikasi runtime produksi, serta addendum lanjutan untuk quota history dan recovery deploy.

## Temuan Root Cause

### 1. Direct-browser OTP gagal karena target bridge salah

Audit request produksi menunjukkan pola berikut:

- flow yang berhasil selalu lebih dulu menghantam root router `http://login.home.arpa/`,
- root router itu menjalankan `login.html` MikroTik yang kemudian me-return user ke `/captive` dengan placeholder/context hotspot yang authoritative,
- flow direct-browser yang gagal justru menembak `http://login.home.arpa/login`, sehingga context router yang dibutuhkan untuk recovery tidak pernah terbentuk.

Kesimpulan:

- manual hotspot login masih boleh memakai `/login`,
- tetapi silent bridge/probe untuk recovery context harus menjaga root router, bukan memaksa semua target lokal menjadi `/login`.

### 2. Auto debt-limit block belum punya hook WA user khusus

Audit runtime dan DB menunjukkan:

- low-quota WA sebelumnya memang sudah pernah terkirim pada sebagian user terdampak,
- block otomatis oleh `policy.block_transition:sync_usage` tetap mencatat `blocked_reason=quota_debt_limit|...`,
- tetapi jalur tersebut belum memanggil template WA khusus debt-limit ke user.

Akibatnya, user bisa melihat status blocked karena debt-limit tanpa menerima pesan penjelasan debt-limit yang sesuai event tersebut.

### C. Root cause bug UI sticky header transparan

#### Gejala

Saat admin membuka dialog `Riwayat Mutasi Kuota` lalu scroll ke bawah, header tabel `Waktu / Event / Perubahan / Quota / Detail` tampak tembus pandang. Konten row pertama terlihat menyatu di bawah header sticky, sehingga tabel tampak rusak dan sulit dibaca.

#### Penyebab

Header sticky sudah memakai `position: sticky`, tetapi shell tabel belum punya background dan stacking context yang cukup kuat. Pada kombinasi `VTable` + `AppPerfectScrollbar`, sel header ikut menempel di atas konten body tanpa lapisan opaque yang benar-benar menutup row di bawahnya.

#### Solusi

Dialog admin history sekarang memakai beberapa guard visual sekaligus:

- `history-table-shell` diberi background solid, radius, dan `isolation: isolate`,
- tabel dipaksa `border-collapse: separate` agar sticky cell tidak mewarisi artefak collapse,
- `thead`, `tr`, dan `th` diberi background opaque yang sama,
- sticky `th` diberi `z-index` lebih tinggi, `background-image` solid, dan shadow bawah agar pemisahan header dari body tetap jelas.

Hasilnya, header sticky tetap solid ketika scroll panjang dan tidak lagi memperlihatkan teks row yang ada di bawahnya.

### D. Insiden deploy recreate di produksi

#### Gejala saat deploy

Deploy awal dengan pola `down` lalu `./deploy_pi.sh --recreate` tidak selesai. Stack berhenti setelah `db` dan `redis` hidup, sementara migrate gagal dengan error:

```text
sqlalchemy.exc.ProgrammingError: (psycopg2.errors.DuplicateColumn)
column "last_hotspot_host_id" of relation "user_devices" already exists
```

#### Root cause

Schema produksi ternyata sudah memiliki kolom hasil migrasi `20260315_add_user_device_host_tracking`, tetapi preflight drift lama di `deploy_pi.sh` menilai revision DB sebagai drift hanya karena `current_rev` tidak termasuk rantai `20260302_*`.

Logika lama:

- membaca target drift di rantai `20260302_*`,
- melihat `current_rev=20260315_add_user_device_host_tracking`,
- karena `current_rev not in CHAIN`, script salah menganggap ini drift yang harus di-stamp ke `20260302_alter_public_update_submission_role_fields`,
- migrate berikutnya mencoba menjalankan migration `20260315_add_user_device_host_tracking` lagi,
- hasilnya `DuplicateColumn`.

Dengan kata lain, masalahnya bukan migration baru dari fitur quota history. Masalahnya adalah preflight auto-stamp lama terlalu agresif dan bisa me-rewind revision turunan yang sebenarnya sudah lebih baru dari chain target.

#### Recovery yang diterapkan

Recovery produksi dilakukan dengan langkah berikut:

1. verifikasi `alembic_version` dan schema live,
2. konfirmasi bahwa kolom `last_hotspot_host_id` dan `last_hotspot_uptime_seconds` memang sudah ada di `user_devices`,
3. `flask db stamp 20260315_add_user_device_host_tracking`,
4. jalankan `flask db upgrade` ulang secara idempotent,
5. start ulang backend, worker, beat, dan frontend,
6. health check publik ulang via `global-nginx-proxy`.

#### Hardening permanen

`deploy_pi.sh` kemudian diperbaiki agar auto-stamp drift hanya aktif jika:

- `current_rev` kosong, atau
- `current_rev` masih berada di dalam chain `20260302_*` namun tertinggal di belakang target.

Script sekarang tidak lagi me-rewind revision turunan yang lebih baru.

### E. PDF export cleanup

Setelah export PDF quota history hidup di produksi, masih muncul warning WeasyPrint:

```text
Ignored `word-break: break-word`, invalid value.
```

Penyebabnya adalah CSS template PDF quota history dan debt report masih memakai property `word-break: break-word` yang diabaikan oleh WeasyPrint.

Solusi:

- hapus `word-break: break-word`,
- pertahankan `overflow-wrap: anywhere` untuk kebutuhan wrapping.

Template yang dibersihkan:

- `backend/app/templates/quota_history_report.html`
- `backend/app/templates/admin_user_debt_report.html`

Hotfix template juga sudah diterapkan ke container backend produksi agar warning tidak menunggu release berikutnya.
