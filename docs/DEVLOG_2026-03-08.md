# Devlog 2026-03-08 (Unauthorized Recovery Hardening, Mobile Refresh Stabilization, dan Root Cause Host Reset Massal)

Dokumen ini merangkum perubahan kode, validasi CI/CD, deploy produksi, serta hasil investigasi insiden "host router terus ke-reset".

## 1) Ringkasan Eksekusi

### Selesai di kode aplikasi
- Hardening backend agar cleanup hotspot host tidak lagi berjalan broad.
- Sinkronisasi unauthorized periodik tidak lagi menghapus trusted host.
- Stabilisasi layout mobile saat refresh agar tidak kembali ke state salah.
- Hardening deploy `--recreate` agar selalu sinkron env publik.

### Selesai di pipeline
- Commit + push patch utama ke `main`.
- Trigger CI dan manual Docker Publish.
- Perbaikan test backend yang gagal karena ekspektasi lama tidak sesuai policy baru.
- Re-run CI + Docker Publish hingga status sukses.

### Selesai di produksi
- Deploy `--recreate` ke revision terbaru.
- Verifikasi revision backend/frontend/celery berjalan pada commit terbaru.

### Temuan insiden penting
- Aplikasi **bukan** sumber reset host massal.
- Sumber reset host massal berasal dari scheduler/script RouterOS eksternal (`Wartelpas: Status BUKA/TUTUP`) yang mengeksekusi `/ip hotspot host remove [find server="wartel"]`.

## 2) Perubahan Kode Aplikasi

### A) Backend: cleanup host dibatasi hanya jalur recovery unauthorized
File:
- `backend/app/services/device_management_service.py`
- `backend/app/commands/sync_unauthorized_hosts_command.py`

Perubahan:
- Tambah guard `_should_cleanup_hotspot_host_after_login(...)`.
- Cleanup hotspot host setelah login hanya dilakukan jika IP login saat itu benar-benar berada pada list `unauthorized` (recovery path).
- `MIKROTIK_UNAUTHORIZED_EXEMPT_IPS` dan `MIKROTIK_UNAUTHORIZED_BYPASS_IPS` dihormati dalam keputusan cleanup recovery.
- Jalur scheduler `sync-unauthorized-hosts` tidak lagi melakukan trusted-host cleanup periodik.

Implikasi:
- User authorized tidak lagi terkena reset host karena sync periodik.
- Recovery unauthorized tetap bekerja saat user benar-benar keluar dari status unauthorized.

### B) Frontend: stabilisasi refresh mobile
File:
- `frontend/@layouts/stores/config.ts`

Perubahan:
- Inisialisasi breakpoint dari `useMediaQuery` sejak awal.
- Kelas layout memakai `effectiveAppContentLayoutNav` agar mobile selalu konsisten vertical saat refresh.

Implikasi:
- State navbar mobile lebih konsisten pada hard refresh/reload.

### C) Deploy script hardening
File:
- `deploy_pi.sh`

Perubahan:
- Pada mode `--recreate`, `.env.public.prod` kini required (selain `.env.prod`).
- Upload file deploy disatukan dan dipastikan mengirim `.env.public.prod` saat recreate.

Implikasi:
- Menghindari drift konfigurasi public runtime setelah recreate.

## 5) Root Cause Host Reset Massal (Bukan dari App)

Ditemukan di RouterOS:
- `/system/scheduler`:
  - `Wartelpas: Status BUKA`
  - `Wartelpas: Status TUTUP`
- `/system/script`:
  - `Wartelpas_BUKA`
  - `Wartelpas_TUTUP`

Keduanya memuat command:
- `/ip hotspot host remove [find server="wartel"]`

Dampak:
- Saat scheduler harian berjalan, host table dibersihkan massal, terlihat sebagai "host ke-reset semua".

## 6) Blocker Operasional Saat Hotfix Live

Percobaan patch scheduler/script lewat API user aplikasi gagal dengan error policy:
- `failure: user's policy does not allow to edit this script`

Makna:
- Akun MikroTik yang dipakai backend tidak punya hak edit `/system/scheduler` dan `/system/script`.

## 7) Tindakan Lanjutan Wajib (Router Admin)

Dijalankan oleh akun router dengan privilege lebih tinggi:
1. Hapus baris `/ip hotspot host remove [find server="wartel"];` dari scheduler:
   - `Wartelpas: Status BUKA`
   - `Wartelpas: Status TUTUP`
2. Hapus baris yang sama dari script:
   - `Wartelpas_BUKA`
   - `Wartelpas_TUTUP`
3. (Sementara) boleh disable scheduler BUKA/TUTUP sampai patch script selesai diterapkan.

Setelah patch router:
- Monitor satu siklus BUKA/TUTUP.
- Pastikan host table tidak lagi drop massal.
- Pastikan sync unauthorized tetap normal (`hotspot_host_cleanup_removed=0` di log worker).

## 8) Catatan Konsistensi Policy

Policy aplikasi saat ini:
- Unauthorized sync -> mengelola address-list unauthorized + guard parity.
- Login sukses user -> keluarkan IP dari unauthorized.
- Cleanup hotspot host -> hanya untuk recovery unauthorized yang valid.

Artinya:
- Jika masih ada reset host massal, cek jalur eksternal router dulu (scheduler/script/manual ops), bukan hanya backend app.
