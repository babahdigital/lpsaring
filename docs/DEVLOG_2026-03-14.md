## Temuan Produksi Paling Penting

### 1. `10.8.0.x` bukan datang dari Nginx atau app logs

- Audit SSH ke `/home/abdullah/nginx/logs` tidak menemukan trace `10.8.0.x`, `154.30.75.26`, atau `172.16.0.2` sebagai IP request/log access yang diproksikan.

- Kesimpulan: IP `10.8.0.x` / publik itu bukan rewrite Cloudflare/Nginx/backend, tetapi artefak state RouterOS `Hotspot Hosts` yang menyimpan original address lama/asing untuk MAC yang sama.

### 2. Gap nyata yang masih bisa mempengaruhi quota sebelum patch

Sebelum sesi ini, helper `get_hotspot_host_usage_map()` masih memiliki 2 masalah:

- MAC duplikat dipilih berdasarkan selector yang belum cukup memprioritaskan row lokal yang aktif.
- bila row terpilih punya `address` di luar CIDR tetapi `to-address` sudah benar di subnet hotspot, consumer tetap membaca IP original yang salah.

Efeknya:

- quota sync bisa mengikuti row lawas yang bukan sesi aktif,
- parity dan repair IP-binding/DHCP bisa memakai IP yang tidak lagi valid,
- unblock/status sync bisa menghapus/menulis address-list ke IP yang salah.

### 3. Anomali log produksi yang lebih besar: PostgreSQL hampir penuh oleh Celery worker

Audit live PostgreSQL menunjukkan:

- `max_connections = 100`
- snapshot penggunaan saat audit: `1 active`, `90 idle`, `1 idle in transaction`, `5 background`
- breakdown client address:
  - `192.168.16.3` (`hotspot_prod_flask_backend`) = `9 idle`
  - `192.168.16.4` (`hotspot_prod_celery_worker`) = `83 idle` + `1 idle in transaction`

Log worker dalam 24 jam terakhir juga masih menunjukkan `FATAL: sorry, too many clients already` pada jalur task sinkronisasi/settings. Ini menjelaskan error operasional yang dapat mempengaruhi quota update walaupun endpoint HTTP kelihatan sehat sesekali.

Root cause paling kuat di kode: hampir semua task memanggil `create_app()` sendiri pada setiap run, sehingga worker bisa membentuk app/engine/pool baru berkali-kali dalam satu process lifetime.

## Perbaikan Kode Sesi Ini

### Hotspot host selection hardening

`backend/app/infrastructure/gateways/mikrotik_client.py`

- MAC duplikat tidak lagi diperlakukan last-row-wins.
- Selector kini memprioritaskan:
  - `address` yang masuk CIDR hotspot,
  - lalu `to-address` yang masuk CIDR hotspot,
  - lalu state trusted (`bypassed`/`authorized`),
  - lalu freshness (`idle-time` kecil),
  - lalu `uptime`,
  - baru kemudian `bytes_total`.
- Jika original `address` di luar CIDR tetapi `to-address` masuk CIDR hotspot, field `address` yang diekspor ke consumer diisi dengan IP lokal hasil translasi, dan original tetap disimpan sebagai `source_address`.

### Cleanup ghost host diperketat

`backend/app/tasks.py`

- cleanup host di luar CIDR sekarang hanya boleh remove row ghost jika untuk MAC yang sama sudah ada current hotspot host row yang benar di subnet lokal.
- Ini sengaja lebih ketat daripada sekadar ARP/DHCP proof agar cleanup tidak memaksa sistem “berharap” router membangun row baru sesudah row lama dihapus.

### Celery task app cache

`backend/app/tasks.py`

- `create_app()` dibungkus cache per worker-process.
- Tujuan: task worker memakai Flask app/SQLAlchemy engine yang stabil, bukan membuat app baru di setiap task run.
- Harapan operasional: idle PostgreSQL connection dari `hotspot_prod_celery_worker` turun drastis setelah deploy karena engine pool tidak lagi beranak per task.

## Follow-Up 2026-03-14 — Hardening Firewall Live dan Residual Quota Sync

### Hardening firewall live yang sudah diterapkan

Sesudah audit read-only awal, policy RouterOS juga dirapikan langsung di produksi pada `R.Utama`.

Perubahan terpenting:

- `vlanMaster (10.1.1.0/24)` tidak lagi menjadi member aktif interface list `MGT`, sehingga seluruh subnet perangkat WiFi/infrastruktur tidak otomatis mewarisi trust boundary management.
- rule broad `FORWARD: Allow Inter-LAN/VLAN (UniFi etc)` dinonaktifkan.
- allow forward lintas jaringan diganti menjadi whitelist interface sumber tepercaya saja:
  - `zerotier1 -> LOCAL_NETWORKS`
  - `ether13-MGT -> LOCAL_NETWORKS` dan `-> zerotier1`
  - `vlanSupport -> LOCAL_NETWORKS` dan `-> zerotier1`
  - `wireguard-do -> LOCAL_NETWORKS` dan `-> zerotier1`
  - `ether6-Pi -> LOCAL_NETWORKS` dan `-> zerotier1`
- status yang dibatasi (`klient_habis`, `klient_expired`, `klient_inactive`, `klient_blocked`) sekarang punya pasangan rule `allow-local-bypass-server` lalu `drop-local-networks`, sehingga klien restricted tidak bisa bergerak lateral ke subnet lokal selain exception bisnis yang memang diizinkan.

Hasil operasional langsung setelah apply:

- jalur admin/backend ke RouterOS tetap sehat melalui `wireguard-do`
- health endpoint aplikasi tetap hijau
- broad lateral movement antar LAN/VLAN berhasil ditutup tanpa memutus jalur management yang memang diperlukan

### Temuan log setelah hardening firewall

Audit SSH ke `/home/abdullah/nginx/logs` dan stack `/home/abdullah/lpsaring/app` menunjukkan:

- `nginx error.log` hanya berisi restart/reload normal lama; tidak terlihat burst error baru yang mengarah ke dampak hardening firewall pada tanggal 14 Maret.
- ringkasan `access` yang terambil didominasi 5xx/499 historis tanggal 4-5 Maret, bukan lonjakan baru setelah hardening RouterOS.
- scan keyword umum pada `docker compose logs` pasca-hardening tidak menunjukkan exception baru yang menonjol di backend/frontend selain jalur quota sync yang memang sedang diinvestigasi terpisah.

### Residual issue yang masih aktif: quota sync session leak

Audit runtime quota setelah hardening firewall tetap menemukan satu masalah backend yang belum tertutup:

- `pg_stat_activity` masih memperlihatkan `idle in transaction = 1`
- worker quota masih mencatat error berulang:

```text
sqlalchemy.exc.InvalidRequestError: A transaction is already begun on this Session.
```

Signature error konsisten muncul di `sync_hotspot_usage_task`, bukan di flow firewall atau nginx.

Hipotesis yang tervalidasi lewat code inspection:

- sebelum loop per-user dimulai, `sync_hotspot_usage_and_profiles()` masih membaca `application_settings`
- pembacaan itu dapat memicu implicit transaction pada scoped session yang sama
- iteration user pertama lalu masuk ke `with db.session.begin():` saat session sudah aktif, sehingga task gagal per-user secara periodik

### Fix lokal yang disiapkan

Patch lanjutan untuk residual quota sync ini sudah disiapkan dan divalidasi lokal:

- runtime settings quota sync dipreload satu kali lewat helper khusus yang selalu memanggil `db.session.remove()` sebelum loop user dimulai
- lookup settings berulang di dalam loop dikurangi dengan memakai runtime snapshot yang sudah dipreload
- test regresi ditambahkan untuk memastikan scoped session dari pembacaan settings pra-loop tidak lagi bocor ke transaksi per-user

Commit fix lokal:

- `c1244b5c` — `fix: release quota sync settings session`

Validasi lokal yang lulus sebelum rollout:

- `tests/test_tasks_hotspot_usage_sync.py`
- `ruff check app/services/hotspot_sync_service.py tests/test_hotspot_sync_user_error_isolation.py app/tasks.py`

Kesimpulan akhir sesi:
