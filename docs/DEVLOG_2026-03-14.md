# Devlog 2026-03-14 — Audit Ghost Host, Quota Safety, dan Saturasi PostgreSQL

Lampiran wajib:
- [.github/copilot-instructions.md](../.github/copilot-instructions.md)

## Context

Audit lanjutan dilakukan sebelum deploy cleanup host hotspot. Fokus sesi ini:

1. memastikan ghost `/ip/hotspot/host` di luar CIDR tidak merusak quota sync,
2. memastikan cleanup row ghost aman terhadap unauthorized flow,
3. membaca ulang log lama + log produksi (`nginx` dan container app) untuk mencari anomali yang belum tertutup.

---

## Temuan Produksi Paling Penting

### 1. `10.8.0.x` bukan datang dari Nginx atau app logs

- Audit SSH ke `/home/abdullah/nginx/logs` tidak menemukan trace `10.8.0.x`, `154.30.75.26`, atau `172.16.0.2` sebagai IP request/log access yang diproksikan.
- Audit `docker compose logs` pada stack `/home/abdullah/lpsaring/app` juga tidak menemukan hit untuk pola IP tersebut.
- Kesimpulan: IP `10.8.0.x` / publik itu bukan rewrite Cloudflare/Nginx/backend, tetapi artefak state RouterOS `Hotspot Hosts` yang menyimpan original address lama/asing untuk MAC yang sama.

### 2. Gap nyata yang masih bisa mempengaruhi quota sebelum patch

Sebelum sesi ini, helper `get_hotspot_host_usage_map()` masih memiliki 2 masalah:

- MAC duplikat dipilih berdasarkan selector yang belum cukup memprioritaskan row lokal yang aktif.
- bila row terpilih punya `address` di luar CIDR tetapi `to-address` sudah benar di subnet hotspot, consumer tetap membaca IP original yang salah.

Efeknya:

- quota sync bisa mengikuti row lawas yang bukan sesi aktif,
- parity dan repair IP-binding/DHCP bisa memakai IP yang tidak lagi valid,
- unblock/status sync bisa menghapus/menulis address-list ke IP yang salah.

### 3. Anomali log produksi yang lebih besar: PostgreSQL hampir penuh oleh Celery worker

Audit live PostgreSQL menunjukkan:

- `max_connections = 100`
- snapshot penggunaan saat audit: `1 active`, `90 idle`, `1 idle in transaction`, `5 background`
- breakdown client address:
  - `192.168.16.3` (`hotspot_prod_flask_backend`) = `9 idle`
  - `192.168.16.4` (`hotspot_prod_celery_worker`) = `83 idle` + `1 idle in transaction`

Log worker dalam 24 jam terakhir juga masih menunjukkan `FATAL: sorry, too many clients already` pada jalur task sinkronisasi/settings. Ini menjelaskan error operasional yang dapat mempengaruhi quota update walaupun endpoint HTTP kelihatan sehat sesekali.

Root cause paling kuat di kode: hampir semua task memanggil `create_app()` sendiri pada setiap run, sehingga worker bisa membentuk app/engine/pool baru berkali-kali dalam satu process lifetime.

## Perbaikan Kode Sesi Ini

### Hotspot host selection hardening

`backend/app/infrastructure/gateways/mikrotik_client.py`

- MAC duplikat tidak lagi diperlakukan last-row-wins.
- Selector kini memprioritaskan:
  - `address` yang masuk CIDR hotspot,
  - lalu `to-address` yang masuk CIDR hotspot,
  - lalu state trusted (`bypassed`/`authorized`),
  - lalu freshness (`idle-time` kecil),
  - lalu `uptime`,
  - baru kemudian `bytes_total`.
- Jika original `address` di luar CIDR tetapi `to-address` masuk CIDR hotspot, field `address` yang diekspor ke consumer diisi dengan IP lokal hasil translasi, dan original tetap disimpan sebagai `source_address`.

### Cleanup ghost host diperketat

`backend/app/tasks.py`

- cleanup host di luar CIDR sekarang hanya boleh remove row ghost jika untuk MAC yang sama sudah ada current hotspot host row yang benar di subnet lokal.
- Ini sengaja lebih ketat daripada sekadar ARP/DHCP proof agar cleanup tidak memaksa sistem “berharap” router membangun row baru sesudah row lama dihapus.

### Celery task app cache

`backend/app/tasks.py`

- `create_app()` dibungkus cache per worker-process.
- Tujuan: task worker memakai Flask app/SQLAlchemy engine yang stabil, bukan membuat app baru di setiap task run.
- Harapan operasional: idle PostgreSQL connection dari `hotspot_prod_celery_worker` turun drastis setelah deploy karena engine pool tidak lagi beranak per task.

## Kesimpulan Audit Sebelum Deploy

1. `10.8.0.x` bukan berasal dari Cloudflare tunnel, Nginx, atau env app; sumbernya tetap konsisten sebagai stale/original-address artifact di RouterOS Hotspot Hosts.
2. Risiko quota yang paling nyata bukan hanya ghost row, tetapi juga pemilihan row host yang salah dan saturasi koneksi PostgreSQL di Celery worker.
3. Kedua gap tersebut sudah dipatch lokal pada sesi ini, sehingga kandidat deploy berikutnya jauh lebih aman dibanding state sebelum audit.
